## 需求分析
### 教学内容
+ 需求分析的任务：数据需求 + 功能需求
+ 数据字典的编制：数据项、数据结构、数据流、数据存储
+ 方法：访谈、问卷调查、调研现有系统


### 功能需求
1. 员工管理：注册、登录、信息维护
2. 会议室管理：添加、删除、修改会议室信息（管理员）
3. 预约功能：员工选择会议室、时间段提交预约
4. 审批功能：管理员审批预约申请
5. 签到功能：会议开始前30分钟可签到
6. 查询功能：查看会议室状态、个人预约记录、会议日历
7. 统计功能：会议室使用率统计

## E-R模型设计
### 教学内容
+ 实体、属性、联系的识别方法
+ 局部E-R图 → 全局E-R图的合并策略
+ E-R图绘制规范
+ 冲突的解决：属性冲突、命名冲突、结构冲突



### 实体与联系分析
**实体：**

+ 员工 (Employee)
+ 会议室 (MeetingRoom)
+ 预约 (Reservation)
+ 签到 (CheckIn) — 弱实体，依赖于预约



**联系：**

+ 员工 1 : N 预约（一个员工可发起多个预约）
+ 预约 N : 1 会议室（一个预约对应一个会议室）
+ 预约 1 : 1 签到（一个预约对应一次签到，签到为弱实体）



### 概念模型
+ **核心目标**：准确、完整地反映用户业务需求中的数据关系，建立独立于任何具体数据库管理系统的信息模型。
+ **主要产出物**：**E-R 图**（实体-关系图）、概念数据模型。
+ **关注视角**：**业务视角**。只关心业务中存在哪些实体（如“顾客”、“订单”、“商品”），以及实体之间存在什么业务联系（如“顾客下单购买商品”），**不涉及任何技术实现细节**。
+ **设计依据**：需求分析阶段得到的数据流图、业务规则和用户调研结果。
+ **典型特征**：与具体数据库软件（Oracle、MySQL）、硬件或存储方式无关。即使不使用数据库，这个模型也能用来描述业务逻辑。
+ **关键活动**：定义实体、定义属性、确定主键（业务主键）、定义实体间的一对一、一对多、多对多关系。
