# 实验日志 · 表与视图的基本操作

_2026-04-28 · SQL Server 2022 · SSMS 22_

## ❌ 错误记录

### BUG-01 · 列名无效

**触发场景：** 给读者表添加"政治面貌"列时报错

```
消息 207，级别 16，状态 1
列名 '政治面貌' 无效
```

**根本原因：** `USE [图书管理]; GO` 和后续 `ALTER TABLE` 混在一起跑，GO之后数据库上下文没有正确传递  
**解决方案：** 分两次执行，或直接在SSMS左上角下拉框手动切换数据库，不写 `USE`

---

### BUG-02 · CREATE VIEW 语法错误

**触发场景：** 创建/修改视图时报错

```
消息 156，级别 15，状态 1，过程 C_P_N
关键字 'SELECT' 附近有语法错误
```

**根本原因：** `CREATE VIEW` / `ALTER VIEW` 后面跟了 `SELECT * FROM C_P_N` 验证语句，SQL Server要求这类语句必须是批处理中的唯一语句  
**解决方案：** 分两次执行，第一次只跑 CREATE/ALTER VIEW，第二次单独跑 SELECT 验证

---

### BUG-03 · 已有数据的表直接加非空列报错

**触发场景：** 给已有5条数据的读者表直接添加 NOT NULL 列  
**根本原因：** 表内已有数据，新列无法为现有行提供非空值  
**解决方案：** 三步走

```sql
-- 第一步：先加 NULL 列
ALTER TABLE [读者] ADD [政治面貌] char(4) NULL;

-- 第二步：填充默认值（分开跑）
UPDATE [读者] SET [政治面貌] = '群众';

-- 第三步：再改为非空（分开跑）
ALTER TABLE [读者] ALTER COLUMN [政治面貌] char(4) NOT NULL;
```

## ✅ 关键知识点备忘

### CREATE VIEW / ALTER VIEW 的正确姿势

```sql
-- ❌ 错误：混在一起跑
ALTER VIEW C_P_N AS
SELECT ...;
SELECT * FROM C_P_N;  -- 这行会导致整批报错

-- ✅ 正确：分两次执行
-- 第一次
ALTER VIEW C_P_N AS
SELECT ...;

-- 第二次
SELECT * FROM C_P_N;
```

### USE + GO 的正确姿势

```sql
-- ❌ 容易出问题
USE [DingBao];
GO
CREATE VIEW ...  -- 有时候上下文没切过去

-- ✅ 最省事
-- 直接在SSMS左上角下拉框选数据库，不写USE
CREATE VIEW ...
```

### 检查视图是否存在

```sql
SELECT * FROM sys.views WHERE name = 'C_P_N';
-- 有结果 → 视图存在
-- 空结果 → 视图已删除
```

### 跨数据库导出表

```sql
SELECT * INTO mydb.dbo.[读者]
FROM 图书管理.dbo.[读者];
-- 不需要切换数据库，直接写完整路径
```

---

## 📝 总结

这次实验最大的坑就一个：**SQL Server 对批处理边界极其敏感**，`CREATE VIEW`、`ALTER VIEW` 这类语句必须独立执行，混进去任何其他语句就报语法错误。解决办法很简单——**分开跑，用下拉框切库，别迷信 USE + GO**。
